- **Real-shape contract (ОБОВ'ЯЗКОВО — фікстури тестів мають дзеркалити РЕАЛЬНИЙ Claude Code, не припущення):** executable-тести викликають скрипти з рукотворними stdin-фікстурами; помилкове ім'я matcher або хибна форма stdin-JSON пройде весь suite, але хук буде тихо-мертвий / ніколи-не-викликаний у живій сесії. Тому: (1) джерело правди — офіційна документація Claude Code hooks (події `SessionStart` / `PostToolUse`, точні поля stdin: `hook_event_name`, `session_id`, `cwd`, `tool_name`, `tool_input.file_path`, а для SessionStart — `source` ∈ `startup|resume|clear|compact`; matcher-семантика). Звірятися зі skill `plugin-dev:hook-development` та актуальними docs ПЕРЕД тим як вважати задачу done. (2) Фікстури `tests/hooks/run.sh` будуються з ключами саме цієї форми; коментар-посилання на джерело у harness. (3) Acceptance-gate у Task 9 явно крос-чекає matcher-імена (`startup|clear|compact` для SessionStart, `Read|Edit|Write|MultiEdit` для PostToolUse) і форму stdin проти документації.

**Мета:** `hooks/lib/discover.sh`, `hooks/lib/version-gate.sh`, і каркас `tests/hooks/run.sh` з tmp-фікстурами. Це база для Tasks 2-4.

### Що зробити
1. Створити `tests/hooks/run.sh` (executable) — harness:
   - Функція `make_fixture` → `mktemp -d`, у ньому `git init`, `CLAUDE.md` з секцією `## Wiki` + backtick-pointer `` `docs/wiki` ``, `docs/wiki/{index.md,schema.md,.usage.json}`. `schema.md` містить `wiki_version: "4.0"`. `index.md` — короткий валідний контент. `.usage.json` = `{}`.
   - Гелпери: `assert_eq`, `assert_contains`, `assert_file_unchanged` (sha до/після), лічильник pass/fail, ненульовий exit при фейлі.
   - `CLAUDE_PROJECT_DIR` на фікстуру; stdin-JSON через heredoc.
   - Прибирання tmp у trap.
2. `hooks/lib/discover.sh` — bash-функція `discover_wiki <start_dir>` (spec «API — hook-скрипти», `discover.sh`):
   - Стартова точка: аргумент, fallback `$CLAUDE_PROJECT_DIR`, fallback `pwd`. Без stdin-парсингу.
   - **Walk-up root (межа обходу):** git-корінь (`git -C <start> rev-parse --show-toplevel`); **поза git-репо (немає toplevel) — межа = `$CLAUDE_PROJECT_DIR`, fallback стартова точка; НІКОЛИ `/`** (інакше boundary-guard знеструмлюється й malicious pointer може дотягнутися до `/etc/passwd` — spec ризик 8). Walk-up зупиняється щойно досягнуто цієї межі, вище не підіймається.
   - На кожному рівні (від start_dir до межі включно) шукати `CLAUDE.md`/`AGENTS.md`/`GEMINI.md`, у ньому секцію `## Wiki` + перший backtick-шлях.
   - **Резолюція шляху:** backtick-шлях відносний → резолвити відносно **каталогу конфіг-файла** (не `pwd`); абсолютний → as-is (spec, edge «відносний пойнтер + підкаталог»).
   - **Boundary-guard (ОБОВ'ЯЗКОВО, застосовується до РОЗВ'ЯЗАНОГО `index.md`, НЕ до каталогу-кандидата):** guard мусить canonicalizeувати саме той файл, що піде в контекст LLM — `realpath "{кандидат}/index.md"` (а не лише `realpath {кандидат}`). Причина: перевірка каталогу проходить, а `{кандидат}/index.md` може бути симлінком на `/etc/passwd` поза коренем (dir-level check його не ловить). Результат `realpath "{кандидат}/index.md"` мусить бути строго під `realpath` кореня walk-up (`{root}/…`); інакше кандидат відхиляється, у stderr `[wiki-hook] pointer поза межами репо, ігнорую: {candidate}`, discovery продовжує walk-up / fallback. **`realpath` викликати з придушеним stderr (`2>/dev/null`)** для неіснуючих шляхів (missing-file realpath не має засмічувати hook stderr); порожній результат realpath → кандидат відхиляється як out-of-bounds.
   - Кандидат валідний якщо `realpath "{кандидат}/index.md"` існує **І** пройшов boundary-guard вище.
   - **Fallback `docs/wiki/index.md` (dir конфіг-файла / start_dir / git-корінь) проходить ТОЙ САМИЙ boundary-guard на розв'язаному `index.md`** — не bypass: `realpath "{fallback}/index.md"` мусить бути строго під коренем (інакше `docs/wiki/index.md → /etc/passwd` або `docs/wiki → /tmp/outside` через симлінк проскочить повз guard). Fallback, що не пройшов guard, відхиляється як out-of-bounds (порожній stdout).
   - Друкує абсолютний `{wiki}` або нічого. Без конфлікт-резолюції кількох вікі, без legacy-детекту.
3. `hooks/lib/version-gate.sh` — bash-функції (spec `version-gate.sh`):
   - **Парсинг версії ЛИШЕ з YAML-frontmatter, не substring-match по всьому файлу:** читати `{wiki}/schema.md` ТІЛЬКИ у межах першого frontmatter-блоку (від першого рядка-`---` до наступного рядка-`---`); у цьому блоці знайти рядок, що точно матчить ключ `wiki_version` зі значенням `"4.0"` (anchored: `^wiki_version:[[:space:]]*"4.0"[[:space:]]*$`, після strip). Причина: legacy/зіпсований `schema.md`, який лише **згадує** рядок `wiki_version: "4.0"` у прозі, прикладі чи migration-log нижче frontmatter, НЕ повинен пройти gate; або відсутній/інший frontmatter → gate fail. Реалізація: awk/sed витягує блок між першою парою `---`, далі точний grep по ключу. Файл без frontmatter (нема закривального `---`) → версія не визначена → gate fail.
   - `wiki_writable <wiki_dir>` → `exit 0` ЛИШЕ якщо frontmatter `{wiki}/schema.md` містить `wiki_version: "4.0"` (за правилом парсингу вище) **І** `{wiki}/.usage.json` уже існує; інакше non-zero.
   - `wiki_bootstrappable <wiki_dir>` → true коли schema-версія поточна, а `.usage.json` ще нема (дозвіл на створення сайдкара з нуля на свіжій-current вікі).
   - Поточну версію `"4.0"` тримати однією константою вгорі (щоб майбутній bump — одне місце).

### Тести (пишуться ПЕРШ у `tests/hooks/run.sh` — TDD)
- **discover.sh:** знаходить за pointer; walk-up із підкаталогу; boundary (не вище git-кореня); нема pointer + нема `docs/wiki/` → тиша (порожній stdout); битий pointer + існуючий `docs/wiki/` → fallback; **pointer поза межами** (`../../etc`, абсолютний `/etc`, симлінк `{repo}/w`→`/tmp/outside` з `index.md`) → відхилено, stdout порожній/fallback, нічого поза коренем не друкується; **симлінк-escape через сам `index.md`** — валідний каталог-кандидат `{repo}/docs/wiki/` (dir проходить boundary), але `{repo}/docs/wiki/index.md → /etc/passwd` (симлінк файла на out-of-bounds ціль) → guard на РОЗВ'ЯЗАНОМУ `index.md` відхиляє, `/etc/passwd` НЕ друкується; **симлінк-escape через fallback dir** — нема pointer, fallback `{repo}/docs/wiki` є симлінком на `/tmp/outside` з `index.md` всередині → fallback boundary-guard на `realpath docs/wiki/index.md` відхиляє, stdout порожній; **відносний pointer + start_dir у підкаталозі** → резолвиться відносно каталогу конфіг-файла, вікі знайдено; **поза git-репо** (фікстура без `git init`) + `CLAUDE.md` з pointer `../../etc/passwd` та `CLAUDE_PROJECT_DIR` на фікстуру → walk-up НЕ підіймається вище `$CLAUDE_PROJECT_DIR`, out-of-bounds pointer відхилено, `/etc/passwd` НЕ друкується; realpath на неіснуючий шлях → без stderr-шуму.
- **version-gate.sh:** `schema.md` frontmatter `4.0` + `.usage.json` є → `wiki_writable` true; `.usage.json` відсутній → `wiki_writable` false, `wiki_bootstrappable` true; `schema.md` frontmatter не `4.0` → обидва false; `schema.md` відсутній → обидва false; **frontmatter-only match:** `schema.md`, де frontmatter НЕ містить `wiki_version` (або має інше значення), але рядок `wiki_version: "4.0"` присутній у ТІЛІ документа (проза / приклад / migration-log нижче закривального `---`) → обидва false (substring-match не повинен проходити); `schema.md` зовсім без frontmatter-блоку (нема пари `---`) → обидва false.

   - **Crash/interrupt recovery (ОБОВ'ЯЗКОВО для mkdir-fallback — інакше deadlock назавжди):** щойно `mkdir` lock-каталогу вдався, писати PID всередину (`echo $$ > "$LOCKDIR/pid"`) і поставити trap, що прибирає lock ПОВНІСТЮ. **Trap мусить спершу видалити pid-файл, тоді сам каталог** — `rmdir` фейлить на непорожньому каталозі («Directory not empty»), тож голий `rmdir "$LOCKDIR"` НІКОЛИ не звільнить lock (у ньому лежить pid) → вічний deadlock. Правильно: `trap 'rm -f "$LOCKDIR/pid" 2>/dev/null; rmdir "$LOCKDIR" 2>/dev/null' EXIT INT TERM` (або еквівалентно `rm -rf "$LOCKDIR"`). Порядок команд у trap: спочатку прибрати вміст, потім каталог.
   - **Stale-lock detection у retry-циклі — ЛИШЕ за мертвим/нечитабельним PID, БЕЗ mtime-крадіжки:** якщо `mkdir` не вдався, форсовано прибрати lock (`rm -f "$LOCKDIR/pid"; rmdir "$LOCKDIR"`, потім повторити `mkdir`) **ТІЛЬКИ** коли PID-власника відсутній/нечитабельний АБО процес мертвий (`kill -0 <pid>` фейл). **Жодного unconditional mtime-OR-клозу:** живий власник (heavy disk I/O, повільний `python3`-старт, CI на повільному диску), що пише `settings.json` довше за timeout, НЕ повинен втратити lock — інакше повертається саме той lost-update race на `~/.claude/settings.json`, від якого lock і захищає (а exit-trap первісного власника ще й прибере lock нового). Якщо PID живий — другий процес ЧЕКАЄ (коротка пауза й повтор до overall-timeout), не краде. mtime використовувати лише як допоміжний сигнал для випадку, коли pid-файл відсутній зовсім (crash між `mkdir` і `echo $$`): нема pid-файла І каталог старший за timeout → вважати stale. Наявний живий pid завжди має пріоритет: не прибирати.
   - `flock`-гілка вивільняється ядром на будь-якому exit процесу — trap потрібен ЛИШЕ для mkdir-fallback.
2. Read `~/.claude/settings.json` **під локом, безпосередньо перед merge** (не кешований раніший знімок). Нема → `{}`. **Битий JSON → СТОП з помилкою, нічого не писати.**

Причина: користувач може згрупувати під одним matcher-entry і наш wiki-хук, і власний невідповідний хук (один elem entry з `hooks[]` на дві команди) — видалення «entry цілком» знесе й користувацький хук.
